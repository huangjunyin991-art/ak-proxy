# Agent & Commit Conventions

## Git 提交信息（中文）规范

统一用两段式（标题 + 可选正文），目标是「一眼看懂改了什么 + 为什么改」。

### 标题格式

- `修复(<范围>)：<一句话概述>`
- `新增(<范围>)：<一句话概述>`
- `优化(<范围>)：<一句话概述>`
- `重构(<范围>)：<一句话概述>`
- `调整(<范围>)：<一句话概述>`
- `杂项(<范围>)：<一句话概述>`

要求：
- `<范围>`：必填，建议用模块名（如 `im`、`admin`、`server`、`proxy` 等）。
- `<一句话概述>`：不超过 30 字，优先写「结果/目的」，避免堆砌实现细节。

### 正文（可选）

- 用 1～3 行说明「为什么」和「关键行为变化/风险点」。
- 不写无意义流水账。

示例：

- `修复(im)：双击聊天标题打开调试弹窗`

正文：
- `提供 cookies/localStorage/runtime 快照，便于排查账号同步问题。`
- `弹窗内提供复制与手动触发同步。`


另外任何时候都要注意：当我提出让你实现某项功能的时候，要尽可能可能模块化，松耦合，最好模块可以独立运作，缺失了此模块系统只是缺少了部分功能但不至于崩溃，在恰当的情况使用设计模式，代码要易于理解，可拓展性要高！像架构师一样思考问题并向我说明你的设计思路得到我的认可之后才可以开始写代码！而且不同模块要用不同文件夹防止混乱，git要在我的明确要求之下才做，而且我如果取消了就必须立刻停止询问我原因而不是继续坚持执行，当然不止git，对任何命令都是如此。
